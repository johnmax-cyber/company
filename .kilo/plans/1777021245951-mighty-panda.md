# Plan: Fix Product Loading on Vercel Deployment

## Problem Summary
Products not rendering on `https://company-cyber.vercel.app/shop.html`. Page shows "0 products — No products found" and stays in "Loading products..." state.

## Root Cause Analysis

### 1. Dual Product Loading Logic (Conflict)
The `shop.html` page has **TWO separate product loading mechanisms** that conflict:

1. **`main.js` `fetchProducts()`** (lines 73-103):
   - Tries Supabase first, falls back to hardcoded `fallbackProducts` array
   - Does NOT try to fetch from `products.json`
   - Renders using `main.js`'s `renderProducts()` (simple cards with icons)

2. **`shop.html` inline `loadProducts()`** (lines 149-187):
   - Tries Supabase first, falls back to `fetch('/products.json')`
   - Renders using inline `renderProducts()` (rich cards with images)
   - **This is the correct one for `shop.html`**

Both run on `shop.html` via separate `DOMContentLoaded` listeners, causing race conditions.

### 2. Fixed Path Issues (Already Applied)
The following fixes were already applied in previous edits:

- `shop.html` line 179: Changed `fetch('products.json')` → `fetch('/products.json')` ✅
- `assets/js/main.js` line 629: Changed `fetch('products.json')` → `fetch('/products.json')` ✅

### 3. Config.js Placeholder Values
- Root `config.js` has placeholder values: `YOUR_SUPABASE_URL_HERE`, `YOUR_ANON_KEY_HERE`
- On Vercel, `vercel.json` rewrites `/config.js` to `/api/config.js` which serves env var values
- If Supabase env vars are not set on Vercel, the fallback to `products.json` should trigger

## Proposed Fix

### Primary Fix: Prevent `main.js` from Rendering on Shop Page
The `main.js` `init()` function should NOT render products on `shop.html` since the inline script handles it with richer UI.

**File: `assets/js/main.js`**
- Modify `init()` function (lines 105-130) to skip rendering on `shop.html`:

```javascript
async function init() {
  renderNavbar();
  renderFooter();
  
  const path = window.location.pathname;
  
  // Fetch products from Supabase (for non-shop pages)
  await fetchProducts();
  
  // Only render with main.js logic for non-shop pages
  if (!path.includes('shop.html')) {
    if (path.includes('shop.html') || path === '/' || path.endsWith('/') || path.includes('index.html')) {
      renderProducts(products);
      setupFilters();
      setupSearch();
    }
  }
  
  if (path.includes('product.html')) {
    loadProductDetail();
  }
  if (path.includes('cart.html')) {
    renderCart();
  }
  if (path.includes('admin.html')) {
    renderAdmin();
  }
  setupMobileMenu();
  updateCartCount();
}
```

Wait, that logic is confusing. Let me simplify:

```javascript
async function init() {
  renderNavbar();
  renderFooter();
  
  const path = window.location.pathname;
  
  await fetchProducts();
  
  // Don't use main.js rendering for shop.html (it has its own inline logic)
  if (!path.includes('shop.html')) {
    if (path === '/' || path.endsWith('/') || path.includes('index.html')) {
      renderProducts(products);
      setupFilters();
      setupSearch();
    }
  }
  
  // ... rest of the function
}
```

Actually, looking more carefully, the original code was:
```javascript
if (path.includes('shop.html') || path === '/' || path.endsWith('/') || path.includes('index.html')) {
  renderProducts(products);
  setupFilters();
  setupSearch();
}
```

So for `shop.html`, it WOULD call `renderProducts(products)`. This is the problem - `main.js` shouldn't render on `shop.html`.

### Correct Fix

**File: `assets/js/main.js`** (line 114)
Change:
```javascript
if (path.includes('shop.html') || path === '/' || path.endsWith('/') || path.includes('index.html')) {
```
To:
```javascript
if (path === '/' || path.endsWith('/') || path.includes('index.html')) {
```

This prevents `main.js` from rendering products on `shop.html`, allowing the inline script's richer rendering to work.
